1. RDMS의 대표적인 설계 방법 (프로젝트 생명주기)
    1) E-R 모델을 이용한 설계
        - 1단계: 요구사항 분석 
            - 요구사항 명세서
        - 2단계: 개념적 설계
            - 개념적 스키마 (E-R 다이어그램)
        - 3단계: 논리적 설계
            - 논리적 스키마 (릴레이션 스키마)
        - 4단계: 물리적 설계
            - 물리적 스키마
        - 5단계: 구현 (개발)
            - SQL문 작성, DBMS에서 실행하여 DB 생성
    2) 정규화를 이용한 설계

2. 설계 1단계: 요구사항 분석
    1) 요구사항을 수집하고 분석하여 업무에 필요한 데이터가 무엇인지,
       그 데이터에 어떤 처리가 필요한지 등을 고려
        - 주요 사용자의 범위 결정
        - 수행하는 업무 분석
        - 인터뷰, 설문조사, 관련 문서 분석 등 방법을 통해 요구사항 수집

3. 설계 2 단계: 개념적 설계
    1) DBMS에 독립적인 개념적인 스키마 설계
    2) 요구사항 분석 결과물을 개념적 구조로 표현
        E-R 모델을 주로 사용 (E-R 다이어그램)
    3) 주요 작업
        - 중요한 개체 추출
            ex) 병원 DB 개발에 필요한 개체
                - 병원 운영에 필요한 사람: 환자, 의사, 간호사 등
                - 병원 운영에 필요한 사물: 병실, 수술실, 의료 장비 등
            - 찾아낸 명사를 개체와 속성으로 분류
                ex) 이젠 마트에 회원으로 가입하려면 회원아이디, 비밀번호, 이름, 나이, 직업을 입력
                    가입한 회원에게는 등급과 적립금이 부여된다
        - 개체 간의 관계
            - 개체 간의 의미있는 연관성
            - 요구사항 문장에서 개체 간의 연관성을 의미있게 표현한 동사를 찾는다
                - 1:1, 1:n, n:n

4. 데이터 모델링(Modeling)
    1) 단순화시켜 표현하는 것
    2) 명확하게 하는 것
    3) 추상화된 반영

5. 모델링의 세가지 관점
    1) 데이터 관점 (Data,What)
        - 업무가 어떤 데이터와 관련이 있는지, 데이터간의 관계는 무엇인지
    2) 프로세스 관점 (Process, How)
        - 업무가 실제 하고 있는 일은 무엇인지, 무엇을 해야하는지
    3) Data vs Process 관점
        - 업무가 처리하는 일의 방법에 따라 데이터는 어떻게 영향을 받고 있는지

6. 데이터 모델링의 3가지 요소
    1) 업무가 어떤것(Things)
    2) 어떤 것이 가지는 성격(Attribute)
    3) 업무가 관여하는 어떤 것과의 관계(Relationship)

7. 데이터 모델링 언어
    1) 어떤 것(Things)
        - 엔티티(Entity), 인스턴스(instance)
    2) 어떤 것의 성격
        - 속성(Attribute), 속성값(Attribute Value)
    3) 어떤 것 간의 관계
        - 관계(Relationship), 페어링(Paring)

8. 데이터 모델링 작업 순서
    1) 엔티티를 그린다
    2) 엔티티를 적절하게 배치한다
    3) 엔티티간 관계를 설정한다
    4) 관계명을 기술한다
    5) 관계의 참여도, 필수 여부를 기술한다
    
9. Barker 표기법
    1) Richard Barker 등에 의해 개발
    2) 이후 지속적으로 개선되어, 오라클 사에서 기본 표기법으로 채택해서 사용하고 있음
    3) 엔터티 (테이블)
       식별자 (기본키), 외래키 (참조무결성제약조건)
       속성   (컬럼)
       관계   (제약조건) : 엔터티와 엔터티 사이의 관계
